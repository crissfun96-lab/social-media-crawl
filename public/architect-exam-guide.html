Domain 1: Agentic Architecture & Orchestration (27%)

Heaviest Domain

This domain carries the most weight. Master these 6 patterns and you control over a quarter of the exam.

Pattern 1 Prompt Chaining

Output of one step feeds the next. A linear pipeline where each step transforms or enriches data sequentially.

Extract
→
Validate
→
Transform
→
Store

When to use: Steps are sequential and dependent. Each step needs the output of the previous one. You can add "gates" between steps to verify quality before proceeding.

Example: Extract text from PDF → Validate format → Transform to structured JSON → Store in database.

Pattern 2 Routing

Classify input first, then route to a specialized handler. A dispatcher pattern that ensures the right tool handles the right job.

Classifier
→
Handler A
Handler B
Handler C

When to use: Different input types need fundamentally different processing. The classifier is lightweight; the handlers are specialized and expensive.

Example: Customer message arrives → Classify as billing / technical / general → Route to specialized prompt + tools for that category.

Pattern 3 Parallelization

Multiple Claude calls run simultaneously, results are aggregated. Two sub-patterns: sectioning (split task into parts) and voting (same task, multiple perspectives).

Task
→
Worker 1
Worker 2
Worker 3
→
Aggregate

When to use: Subtasks are independent and can be defined upfront. Key: you know exactly what each worker should do before starting.

Example (sectioning): Analyze code repo → Worker 1 reviews security, Worker 2 reviews performance, Worker 3 reviews types → Merge reports.

Example (voting): Same essay graded by 3 Claude instances → majority score wins.

Pattern 4 Orchestrator-Workers

A central orchestrator agent dynamically delegates to worker agents. The orchestrator decides what needs to happen at runtime.

Orchestrator
→
Worker A
Worker B
Worker ?
→
Synthesize
Key Distinction vs. Parallelization

In parallelization, you define the workers upfront. In orchestrator-workers, the orchestrator decides at runtime what workers to spawn and what they should do. The orchestrator adapts to the input.

When to use: Task breakdown is not known upfront. Complex requests that need dynamic decomposition.

Example: "Refactor this codebase" → Orchestrator analyzes repo structure → Spawns workers per module → Workers refactor → Orchestrator merges and validates.

Pattern 5 Evaluator-Optimizer

A feedback loop between a Generator and an Evaluator. The generator produces output, the evaluator critiques it, and the generator improves.

Generator
→
Output
→
Evaluator
→
Feedback
→
Generator
↺ Repeat until quality threshold met

When to use: Quality matters more than speed. Clear evaluation criteria exist. You are willing to trade latency for output quality.

Example: Generate marketing copy → Evaluator checks brand voice, grammar, CTA → Feedback: "too formal, add urgency" → Generator revises → Repeat.

Pattern 6 Autonomous Agent

The most powerful and most risky pattern. An agent with tools and memory that runs a Think → Act → Observe loop until the task is done.

Think
→
Act (Tool)
→
Observe
→
Think

↺ Loop until done or exit condition

Required Safeguards
  • Exit conditions — max iterations, timeout, success criteria
  • Token budgets — prevent runaway costs
  • Human-in-the-loop (HITL) — approval gates for destructive actions
  • Sandboxing — limit what tools the agent can access

When to use: Open-ended tasks where the path to solution is uncertain. The agent needs to explore, backtrack, and adapt. Always pair with safeguards.

The "Keep It Simple" Principle

The exam explicitly tests when NOT to use agents. Key rules:

  • Use the simplest pattern that solves the problem
  • A single Claude API call with a good prompt beats a complex multi-agent system for most tasks
  • Prompt chaining before parallelization. Parallelization before orchestrator-workers.
  • Autonomous agents are the last resort, not the default
  • More agents = more latency, more cost, more failure modes

Exam trap: A scenario describes a straightforward task, and one answer option proposes an elaborate multi-agent architecture. The correct answer is often the simpler approach.

Domain 2: Tool Design & MCP Integration (18%)

MCP: The 3 Primitives

Tools

Functions the model can call. They execute actions and return results. Think: API calls, database queries, file operations.

Resources

Data sources the model can read. They provide context without side effects. Think: files, database records, API responses.

Prompts

Reusable prompt templates. They standardize how the model interacts with tools and resources. Think: parameterized system prompts.

MCP Architecture

Host (IDE/App)
→
Client
→
Server
→
External System
  • Host: The application (Claude Desktop, IDE extension, custom app)
  • Client: Maintains 1:1 connection with a server, handles protocol
  • Server: Exposes tools, resources, and prompts to the client
  • External System: Databases, APIs, file systems, etc.

Transport Layer

stdio — Local

Standard input/output. Server runs as a child process. Best for local tools (filesystem, CLI). Low latency, simple setup.

SSE / HTTP — Remote

Server-Sent Events or HTTP streaming. Server runs remotely. Best for shared services, cloud tools, multi-user setups.

Tool Calling Flow (5 Steps)

1. User Prompt
→
2. Claude Selects Tool
→
3. App Executes
→
4. Result to Claude
→
5. Final Response

Claude never executes tools directly. Your application code sits between Claude and the tool execution. This is the "human-in-the-loop" by default.

Tool Choice Modes

ModeBehaviorUse When
autoClaude decides whether to call a tool or respond directlyGeneral use (default)
anyClaude must call at least one toolForcing tool use for structured extraction
tool (specific)Claude must call the named toolWhen you know exactly which tool is needed

Tool Design Best Practices

  • Atomic: Each tool does one thing well
  • Clear descriptions: Claude reads the description to decide when to use it
  • JSON Schema validation: Define input schemas with types, required fields, enums
  • Structured results: Return JSON, not freeform text
  • Graceful errors: Return error objects, never throw unhandled exceptions
  • Minimal side effects: Prefer read-only tools; flag destructive tools clearly
Anti-Patterns (Expect Exam Questions)
  • God tools: One tool that does everything — confuses Claude's selection logic
  • No descriptions: Claude cannot select the right tool without clear descriptions
  • No input validation: Garbage in, garbage out — always validate with JSON Schema
  • No error handling: Unhandled errors crash the agentic loop
  • Exposing raw internals: Tool names/schemas that leak implementation details

Domain 4: Prompt Engineering & Structured Output (20%)

System Prompt Design

  • Role definition: Tell Claude who it is and what it can/cannot do
  • Constraints: Output format, length limits, forbidden topics
  • Context: Background information the model needs for every response
  • Instructions: Step-by-step rules for processing

Best practice order: Role → Context → Instructions → Constraints → Output Format

Few-Shot Prompting

Provide 2–5 examples of input/output pairs before the actual request. Claude learns the pattern from examples.

  • 2–3 examples: Sufficient for most tasks
  • 4–5 examples: Complex or ambiguous tasks
  • Include edge cases in examples to show handling of unusual inputs
  • Use consistent formatting across all examples

Chain-of-Thought (CoT)

Ask Claude to think step by step before giving the final answer. Improves accuracy on reasoning-heavy tasks.

  • Simple CoT: "Think step by step before answering."
  • Structured CoT: "First, identify X. Then, analyze Y. Finally, conclude Z."
  • Works best for: math, logic, multi-step analysis, code debugging
  • Less useful for: simple factual recall, creative writing

Structured Output Techniques

JSON Mode

Force Claude to return valid JSON. Use a system prompt that specifies the exact JSON schema expected.

Key: Describe the schema in the prompt. Claude will conform to it. Validate the output against the schema in your application code.

Tool Use for Extraction

Define a "tool" with the exact schema you want, then force Claude to call it using tool_choice: any. The tool's input IS your structured data.

Key: The tool doesn't need to do anything. You extract the arguments. Most reliable method for structured output.

XML Tags

Use XML-style tags to structure output: <answer>, <reasoning>, <sources>. Claude naturally respects XML boundaries.

Key: Great for mixing structured and unstructured content. Easy to parse with regex.

Prefilling

Start Claude's response with specific text by pre-populating the assistant message. Forces the model to continue from that point.

Key: Prefill with {" to force JSON. Prefill with <result> to force XML. Powerful steering technique.

Exam Focus

Expect questions asking which structured output technique is best for a given scenario. Tool use for extraction = most reliable. JSON mode = simplest. Prefilling = most flexible steering. XML tags = best for mixed content.

Domain 5: Context Management & Reliability (15%)

Context Window: 200K Tokens (All Models)

All Claude models support a 200K token context window. But filling it completely degrades performance. Strategies to manage context effectively:

Context Strategies

StrategyHow It WorksBest For
SummarizationPeriodically summarize conversation history and replace full history with summaryLong conversations, chat assistants
Sliding WindowKeep only the most recent N messages, drop oldestStreaming applications, real-time chat
RAGRetrieve relevant chunks from external store, inject into contextLarge knowledge bases, documentation
Prompt CachingCache static prefix of prompt; only send dynamic portionRepeated similar requests, cost reduction
Context BudgetingAllocate token limits per section: system prompt, examples, user input, outputProduction systems, predictable costs

RAG Architecture

User Query
→
Embed Query
→
Vector Search
→
Top-K Chunks
→
Inject + Prompt
→
Claude Response
  • Chunking: Split documents into overlapping chunks (512–1024 tokens typical)
  • Embedding: Convert chunks to vectors using an embedding model
  • Retrieval: Find top-K most similar chunks to the user's query
  • Injection: Place retrieved chunks into the prompt as context
  • Generation: Claude answers based on injected context + original query

Prompt Caching

Cost reduction: ~90% on cached portions of the prompt. The API caches the static prefix and only charges full price for the dynamic suffix.

  • System prompt, few-shot examples, and static instructions are cached
  • User's specific query is the dynamic (non-cached) portion
  • Cache has a TTL (time-to-live) — reuse within the window to benefit
  • Mark cache breakpoints with cache_control in the API request

Reliability Patterns

Retry with Backoff

On transient failures (rate limits, timeouts), retry with exponential backoff. Typical: 3 retries, 1s/2s/4s delays.

Fallback Chains

If primary model fails, fall back to secondary. Example: Opus → Sonnet → Haiku. Degrade gracefully.

Guardrails

Input validation (block harmful prompts), output validation (check for PII, hallucinations), content filtering, schema conformance checks.

Human-in-the-Loop (HITL)

Approval gates for high-stakes actions. Confidence thresholds: auto-approve above 95%, flag for review below. Essential for production.

Monitoring & Observability

  • Token usage tracking: Monitor input/output tokens per request to detect drift
  • Latency metrics: P50, P95, P99 response times
  • Error rates: Track by error type (rate limit, timeout, content filter)
  • Quality metrics: User feedback loops, automated eval scores
  • Cost dashboards: Per-endpoint, per-model cost tracking

Sample Questions

Click an answer to check it. These mirror the style and difficulty of real exam questions.

Q1. A team needs to process customer emails by first categorizing them (billing, technical, general), then routing each to a specialized handler. Which agentic pattern is most appropriate?

A. Prompt Chaining
B. Routing
C. Orchestrator-Workers
D. Autonomous Agent
B is correct. The Routing pattern classifies input first and dispatches to specialized handlers. This is exactly classification + dispatch. Orchestrator-Workers would be overkill since the categories are known upfront.

Q2. What is the most reliable method for extracting structured data from unstructured text using the Claude API?

A. Ask Claude to return JSON in a regular prompt
B. Use XML tags to structure the output
C. Define a tool with the desired schema and force Claude to call it using tool_choice
D. Prefill the assistant message with the beginning of the JSON
C is correct. Tool use for extraction is the most reliable method. You define a tool whose input parameters match your desired schema, set tool_choice: any, and extract the structured data from the tool arguments. The schema is enforced by the API.

Q3. In the MCP architecture, which component maintains a 1:1 connection with an MCP server?

A. Host
B. Client
C. Server
D. External System
B is correct. In MCP, the Client maintains a 1:1 connection with a Server. The Host (application) may create multiple Clients, each connecting to a different Server. The Server exposes tools, resources, and prompts.

Q4. A Claude Code project has a CLAUDE.md in the project root and another in a subdirectory. A developer also has a global ~/.claude/CLAUDE.md. When Claude Code processes a file in the subdirectory, what happens?

A. Only the subdirectory CLAUDE.md is loaded
B. Only the project root CLAUDE.md is loaded
C. All three are loaded and merged, with subdirectory taking highest precedence
D. The global and project root are loaded, subdirectory is ignored
C is correct. All three CLAUDE.md files are loaded and merged. The hierarchy is: subdirectory (highest) → project root → global (lowest). More specific files supplement and override less specific ones, but all are loaded.

Q5. An application processes thousands of invoices daily using the same system prompt and few-shot examples. Only the invoice content changes. Which technique best reduces API costs?

A. Reduce the number of few-shot examples
B. Use prompt caching to cache the static prefix
C. Switch to a smaller model
D. Use RAG to retrieve relevant examples dynamically
B is correct. Prompt caching caches the static system prompt and few-shot examples, reducing costs by ~90% on those cached tokens. Since thousands of invoices share the same prefix, this is the most impactful optimization.

Q6. A developer wants to build an agent that refactors a large codebase. The specific files and changes needed are not known upfront. Which pattern is most appropriate?

A. Prompt Chaining
B. Parallelization
C. Orchestrator-Workers
D. Evaluator-Optimizer
C is correct. Orchestrator-Workers is the right pattern when the task breakdown is not known upfront. The orchestrator analyzes the codebase first, then dynamically creates and delegates to workers. Parallelization requires knowing the subtasks upfront.

Q7. What are the three primitive types exposed by an MCP server? (Select all that apply)

A. Tools
B. Resources
C. Endpoints
D. Prompts
A, B, and D are correct. MCP servers expose three primitives: Tools (executable functions), Resources (readable data), and Prompts (reusable templates). "Endpoints" is not an MCP primitive.

Q8. A team is building a content generation system where quality is critical. They want Claude to generate blog posts, then have another instance evaluate quality and provide feedback for revision. Which pattern fits best?

A. Routing
B. Prompt Chaining
C. Evaluator-Optimizer
D. Parallelization
C is correct. Evaluator-Optimizer creates a feedback loop: Generator produces content → Evaluator critiques it → Generator revises based on feedback. This loop continues until quality thresholds are met. It trades latency for quality.

Exam Day Strategy

Before the Exam

  • Get 7-8 hours of sleep the night before
  • Quick 15-minute review of the 6 agentic patterns
  • Have water and a quiet environment ready
  • Test your internet connection and browser

During the Exam

  • Read scenarios carefully the first time — multiple questions reference each
  • Answer every question — no penalty for guessing
  • Flag uncertain questions and return to them
  • Manage time: don't spend more than 2 minutes per question

Common Traps to Avoid

Trap 1: Over-Engineering

The question describes a simple task. One answer option proposes a complex multi-agent system. The correct answer is almost always the simpler approach. Remember: "Keep It Simple" is a testable principle.

Trap 2: Orchestrator vs. Parallelization

Both use multiple workers. The key distinction: in Parallelization, workers are defined upfront. In Orchestrator-Workers, the orchestrator decides at runtime. If the question says "the breakdown is not known ahead of time" = Orchestrator.

Trap 3: Confusing MCP Primitives

Tools execute actions. Resources provide data (read-only). Prompts are templates. If the question asks about reading data without side effects = Resource, not Tool.

Trap 4: Config Hierarchy Direction

More specific overrides less specific. Environment variables beat everything. Local settings beat project settings. Subdirectory CLAUDE.md beats project root CLAUDE.md. Don't get the direction backwards.

Trap 5: "Best" vs. "Acceptable"

Multiple answers may work, but the exam asks for the best or most appropriate option. Look for the answer that is simplest, most reliable, and most aligned with documented best practices.

Trap 6: Autonomous Agent as Default

Autonomous agents are powerful but risky. The exam tests whether you know they should be the last resort, not the first choice. Always check if a simpler pattern works first.
